# NESRecomp — отчёт по багам и чеклист исправлений

Анализ файла `nesrecomp.py` (репозиторий `mxvisor/Nesrecomp`, ветка `main`).
Все ссылки на код приведены по состоянию на коммит `c031e88` (или близкий).

Документ структурирован по убыванию серьёзности: сначала баги корректности
(сгенерированный C ведёт себя не как реальный 6502), затем архитектурные
ограничения (статически не покрывается значимая часть кода), затем мелочи.

Методология проверки корректности (критерии успеха L0–L4, автоматизация
сравнения с эмулятором) и стратегии достижения полного покрытия вынесены
в отдельный документ: `nesrecomp-verification.md`.

---

## 1. Баги корректности эмиссии

Эти проблемы означают, что сгенерированный C-код **выполняет программу
неправильно** — поведение расходится с реальным 6502/2A03 даже на коде,
который BFS успешно нашёл.

### 1.1. Двойной `mem_read` в ADC и SBC для memory-режимов

**Где:** `emit_instruction`, ветки `mn == "ADC"` и `mn == "SBC"`.

**Что не так:** функция `rd()` (она же `emit_read`) при memory-режимах
возвращает строку `mem_read(<addr_expr>)`. В сгенерированных строках
текущая реализация подставляет `{rd()}` дважды:

```python
lines += [
    f"{{ uint16_t r = cpu.A + {rd()} + cpu.C;",           # ← rd() первый раз
    " cpu.V = (~(cpu.A ^ " + rd() + ") & ...) >> 7;" if m=="imm"
    else " { uint8_t _v=" + rd() + "; cpu.V = ... }",     # ← rd() второй раз
    ...
]
```

Для `m == "imm"` это безвредно — `rd()` возвращает литерал
(`"0xNN"`), и двойная подстановка просто означает два литерала.
Но для `zp/zpx/zpy/abs/abx/aby/izx/izy` это даёт **два независимых
вызова `mem_read`** в скомпилированном коде.

**Симптом:** для обычной RAM (`$0000–$07FF`) разницы нет — чтение
идемпотентно. Но для memory-mapped I/O чтение имеет побочные эффекты:

| Регистр | Что делает повторное чтение |
| --- | --- |
| `$2002` PPUSTATUS | Сбрасывает write-toggle (latch) PPU + очищает VBlank flag |
| `$2004` OAMDATA | Возвращает байт OAM, но позиция читалки иногда зависит от состояния |
| `$2007` PPUDATA | Инкрементирует VRAM-адрес на 1 или 32 и возвращает буфер |
| `$4015` APU status | Сбрасывает frame interrupt flag |
| `$4016/$4017` controller | Сдвигает shift-регистр контроллера |

Если в игре где-то встретится `ADC $2007` или `SBC $4015`,
сгенерированный код прочитает регистр дважды и повредит состояние
PPU/APU или потеряет байт ввода. Игры редко делают ADC/SBC прямо
с регистров, но через `izy`-адресацию (`ADC ($zp),Y`), где
`$zp` может указывать на $2007, такое возможно.

**Как проверить:** в выводе `<game>_full.c` найти `func_*` с
`ADC` или `SBC` не-immediate-режима и убедиться, что `mem_read`
подставлен один раз через временную переменную.

**Не затрагивает:** `BIT`, `CMP/CPX/CPY`, `ASL/LSR/ROL/ROR memory`,
`INC/DEC` — они уже используют `_t = mem_read(...)` один раз.

### 1.2. Необработанные недокументированные RMW и immediate-опкоды

**Где:** `OPTABLE` содержит `SLO, RLA, SRE, RRA, DCP, ISB, ANC, ALR,
ARR, XAA, SBX, AHX, TAS, SHY, SHX, LAR` (через `_MORE_UNDOC`),
но `emit_instruction` не имеет веток для этих мнемоник.

**Что не так:** при встрече этих опкодов BFS корректно проходит
через них (потому что они в OPTABLE и имеют `size`), но эмиссия
падает в финальный `else`:

```python
else:
    lines += [f"/* UNHANDLED {mn} */"]
```

То есть в сгенерированном C **операция полностью отсутствует** —
есть только присваивание `cpu.PC`, прибавление циклов и комментарий.
Память не модифицируется, регистры не меняются.

**Симптом:** игры, использующие недокументированные опкоды,
ведут себя непредсказуемо. Некоторые коммерческие игры это делают
(Puzznic, Disch's homebrew, ряд хакеров используют их для
оптимизации). Большинство — нет.

Особенно опасны `SLO/RLA/SRE/RRA/DCP/ISB` — это read-modify-write
с побочкой по флагам (используются как «два-в-одном»).
Их отсутствие будет проявляться как «глюки спрайтов» или
«сломанный счёт».

**Как проверить:** `grep "UNHANDLED" generated/<game>_full.c`.

**Альтернативный подход:** если решено не поддерживать undoc, исключить
их из `OPTABLE` целиком — тогда BFS будет останавливаться на них,
а не идти дальше с дырой в логике.

### 1.3. BRK пушит в стек неправильный адрес возврата

**Где:** `emit_instruction`, ветка `mn == "BRK"`.

**Что не так:** на реальном 6502 инструкция BRK однобайтная по опкоду,
**но при срабатывании она пушит `PC + 2`** — пропускает следующий байт
(сигнатуру/код причины прерывания). Текущая реализация:

```python
elif mn == "BRK":
    lines += ["cpu.PC++;",                                # → PC+1
              "stack_push((cpu.PC >> 8) & 0xFF);",
              "stack_push(cpu.PC & 0xFF);",
              "stack_push(get_P() | 0x10);",              # ← без бита U
              ...
```

В стек уходит `PC+1` вместо `PC+2`. Дополнительно, флаги пушатся
с B=1, но без U=1; на реальном 6502 при BRK пушится `P | 0x30`
(оба бита), как в PHP.

**Симптом:** `RTI` после BRK-handler-а вернётся на байт сигнатуры,
который будет декодирован как опкод. Для большинства игр пофиг
(BRK не используется), но Final Fantasy и некоторые другие
RPG используют BRK как software interrupt.

**Связанная проблема:** в `OPTABLE` для `BRK` стоит `size=1`. При BFS
это означает, что обход после BRK продолжается с `pc+1`. Если BRK
не терминатор (а он добавлен в `TERMINATORS`, поэтому BFS останавливается),
то это OK — после BRK код всё равно не обходится. Но если когда-нибудь
будет fall-through логика, размер 2 будет корректнее.

### 1.4. Не учитываются page-cross циклы

**Где:** `emit_instruction`, строка
```python
lines.append(f' g_cpu_cycles += {op.cycles};')
```

**Что не так:** на реальном 6502 ряд инструкций добавляет +1 цикл
при пересечении страницы (high-byte эффективного адреса меняется
после прибавления индекса):

| Инструкция | +1 цикл при cross-page |
| --- | --- |
| `LDA/LDX/LDY abs,X`, `abs,Y` | Да |
| `ADC/SBC/AND/ORA/EOR/CMP abs,X`, `abs,Y` | Да |
| `LDA ($zp),Y` | Да |
| `STA abs,X/Y`, `STA ($zp),Y` | **Нет** — всегда +1 даже без cross |

Текущая таблица `OPTABLE` содержит «базовое» число циклов
без учёта этого +1. Например, `0xBD LDA abs,X` указан как `4`,
но реально 4 или 5.

**Симптом:** PPU/APU будут тикать на чуть меньшее количество циклов,
чем должны. Для игр с raster-эффектами на основе `sprite-0 hit`
(SMB1 status bar) и для MMC3 scanline IRQ это значит «дрожащая
граница раздела» или «срыв скроллинга в определённых местах».
Для DMC IRQ — потенциально неточный триггер прерываний.

Yield-after-every-instruction подход здесь особенно болезнен:
вы платите цену частых yield-ов именно ради точной синхронизации
PPU/APU, но точности нет из-за приближённых циклов.

---

### 1.5. Не учитываются branch-циклы (taken и cross-page)

**Где:** `emit_instruction`, ветка `mn in BRANCH_OPS`.

**Что не так:** branch-инструкции (`BCC/BCS/BEQ/BNE/BMI/BPL/BVC/BVS`)
на 6502 потребляют:
- 2 цикла, если ветка **не взята**;
- 3 цикла, если взята и target на той же странице;
- 4 цикла, если взята и через границу страницы.

Текущий эмит всегда добавляет `g_cpu_cycles += 2`, независимо от
того, взята ли ветка. Это занижение в горячих циклах (а tight
loop на `BNE` — самая горячая конструкция NES-кода) — таймер
PPU отстаёт от реального.

**Симптом:** аналогично 1.4 — рассинхрон с PPU/APU, особенно
в коде ожидания (`waitvbl: BIT $2002 / BPL waitvbl`).

---

### 1.6. PHP/PLP флаги и проверка `set_P`

**Где:** `emit_instruction`, ветки `PHP`, `PLP`, `BRK`.

**Что не так (потенциально):**
- `PHP` корректно пушит `get_P() | 0x30` — оба бита B и U установлены.
- `PLP` делает `set_P(stack_pop())` — это OK, если реализация
  `set_P` в `runner.c` **игнорирует** биты 4 и 5 (B и U не существуют
  как реальные флаги, это синтетика на стороне push). Если `set_P`
  записывает их в `cpu.B` / `cpu.U` поля — могут быть тонкие баги
  при последующих `PHP`.
- `BRK` пушит `get_P() | 0x10` (см. 1.3) — несогласовано с PHP,
  должно быть `0x30`.

**Что проверить:** реализацию `set_P` в `runner.c`/`cpu_interp.c`.
Если она игнорирует bits 4-5 — всё хорошо, кроме согласованности
BRK с PHP.

---

### 1.7. Indirect JMP page-bug на эмуляторной стороне

**Где:** не в `nesrecomp.py`, а в `runner.c` / `cpu_interp.c` —
функция `ind_addr(addr)`.

**Что не так:** на реальном 6502 при `JMP ($XXFF)` процессор читает
low-байт с `$XXFF`, а high-байт с `$XX00` (а не с `$(XX+1)00`).
Это известный аппаратный баг 6502. `ind_addr` должна его воспроизводить.

**Симптом:** если игра случайно или намеренно использует indirect
JMP через адрес с low-байтом FF — на исправленной реализации
получит «не тот» target. На реальной NES получит баговый target.

**Что проверить:** реализацию `ind_addr` в C-коде runner-а.
Аналогично `izx_addr` / `izy_addr` — у них тоже есть нюансы
с zero-page wraparound (low + 1 должно делать `(low+1) & 0xFF`,
а не идти в `$0100`).

## 2. Архитектурные ограничения дискаверера

Эти проблемы **не делают сгенерированный код неправильным**.
Когда BFS не находит код, dispatch-таблица в `default:` вызывает
`cpu_interp_step()`, и интерпретатор корректно исполняет инструкцию.
Корректность сохраняется во всех случаях — речь идёт об **упущенных
возможностях оптимизации**: код, который мог бы исполняться через
рекомпилированные C-функции, идёт через интерпретатор.

Практический эффект зависит от того, насколько yield-after-every-instruction
подход вообще быстрее интерпретатора (см. **2.6**). Если разница
небольшая — упущенное покрытие BFS малозаметно. Имеет смысл сначала
измерить базовый прирост скорости на хорошо покрытой NROM-игре
(см. блок «Тестирование»), и только потом инвестировать в улучшение
покрытия для сложных мапперов.

### 2.1. MMC3: switchable banks идут через интерпретатор

**Где:** `Disassembler.is_switchable` + использование в `_bfs`:

```python
def is_switchable(self, addr: int) -> bool:
    return self.mapper == 4 and 0x8000 <= addr < 0xC000
```

И во всех местах BFS — `if self.is_switchable(tgt): continue`.

**Что происходит:** для MMC3 диапазон `$8000–$BFFF` действительно
переключается runtime, и для одного и того же CPU-адреса там в
разные моменты может быть разный код. Поэтому статически собрать
**одну** функцию по CPU-адресу нельзя. Текущее решение — пропустить
весь этот диапазон, и при вызове в него dispatch уходит в `default:`
→ `cpu_interp_step()`.

**Это работает корректно.** На практике MMC3-игры запускаются и
играются нормально — fallback на интерпретатор написан штатно
именно для таких случаев. Эмпирически проверено на реальных играх.

**В чём собственно ограничение:** в MMC3-играх в switchable банках
часто живёт основная игровая логика (уровни, AI, level handlers).
Эта часть выполняется через интерпретатор, а не через
рекомпилированные функции. Сколько именно теряется производительности —
зависит от того, какой прирост даёт сама рекомпиляция относительно
интерпретатора (см. **2.6**). Если прирост невелик, разница
малозаметна; если значителен, MMC3-игры будут идти ощутимо медленнее
аналогичных NROM-игр.

**Что точно стоит проверить на практике:**
- Бенчмарк FPS на NROM-игре (SMB1) vs MMC3-игре (SMB3) в режиме
  без vsync-капа. Если разница в пределах 10–20% — текущий fallback
  оправдан и стоит сосредоточиться на других пунктах. Если разница
  в разы — есть смысл вкладываться в bank-aware рекомпиляцию.
- Профайл с `perf record` на MMC3-игре: какой процент CPU-времени
  уходит в `cpu_interp_step` vs в `func_*`.

**Bank-aware рекомпиляция** (если измерения покажут, что это окупается):
компилировать по версии функции на банк, имена вида `func_BANK02_8123`,
`func_BANK05_8123` и т.д. Dispatch становится двумерным:
`call_by_address(bank, addr)`, где `bank` берётся из текущего
состояния mapper-а. Это:
- увеличивает размер сгенерированного кода (до N × объём switchable
  области, где N — число банков);
- требует BFS с учётом «текущий банк» (seed = `(bank, addr)`);
- требует, чтобы mapper-state был доступен из dispatch (он и так есть,
  но индексация должна быть быстрой — `static const void* table[N][...]`
  или similar).

**Отдельный вопрос про MMC1/UNROM/CNROM (mapper 1/2/3).** У них тоже
есть switchable слот (16K в `$8000–$BFFF` или `$C000–$FFFF` в
зависимости от режима MMC1), но `is_switchable` сейчас проверяет
только `mapper == 4`. То есть для этих мапперов BFS обходит код
в switchable слоте, считая его статическим — и компилирует его так,
будто там всегда лежит банк, видимый при чтении ROM-а
`Disassembler.prg_read`.

Это **потенциально** опасно: если игра в runtime переключит банк,
вызов в `$8000` попадёт не в `func_8000`, который мы сгенерировали
(он соответствует тому банку, что был в момент discovery), а должен
бы выполнять код другого банка.

Однако твоё наблюдение, что MMC3-игра работает корректно, наводит
на мысль, что либо:
- в `prg_read` для не-MMC3 уже есть какая-то логика, которой я не
  заметил при чтении;
- либо UNROM/MMC1-игры тоже на самом деле работают (надо проверить);
- либо в этих играх логика в switchable слоте дублируется или не
  вызывается часто, и проблема маскируется.

**Проверить на UNROM-игре** (Castlevania 1, Mega Man 1, Contra,
Metal Gear): запустить, поиграть, сравнить с эталоном (FCEUX/Mesen).
Если глюков нет — текущая модель валидна и стоит уточнить
формулировку. Если глюки есть — это реальный баг, аналогичный MMC3,
и `is_switchable` нужно расширить на mapper 1/2.

### 2.2. Inline-data после JSR (PLA/PLA pattern)

**Где:** BFS в `_bfs`:

```python
if op.mnemonic == "JSR":
    tgt = operand & 0xFFFF
    if tgt not in visited and not self.is_switchable(tgt):
        queue.append(tgt)
    ret = pc + 3
    if ret not in visited and not self.is_switchable(ret):
        queue.append(ret)
```

**Что не так:** BFS безусловно добавляет `pc+3` как seed после JSR.
Это предполагает, что подпрограмма всегда возвращает в адрес
сразу после JSR. Но классический паттерн в NES-играх:

```
JSR print_string
.byte "HELLO WORLD", 0
LDA #$00            ; ← сюда реально вернёмся
```

Подпрограмма `print_string` снимает return-адрес со стека через
`PLA / PLA`, читает строку (memory-байты после JSR), вычисляет
конец строки, проталкивает обратно новый адрес и делает `RTS`.

Текущий BFS попытается дизассемблировать байт `'H'` (0x48 = `PHA`)
как опкод, затем `'E'` (0x45 = `EOR $..`) и т.д. Это:
- создаёт «фантомные» функции из строковых данных;
- эти функции могут содержать JSR/JMP в случайные адреса,
  что протягивает мусор по всему графу;
- засоряет dispatch-таблицу и `_full.c`.

**Симптомы:**
- неоправданно большой `_full.c`;
- ложные срабатывания: dispatch попадает в функцию из строки,
  выполняет несколько «инструкций» (бессмысленных, но изменяющих
  регистры/память), затем `runner_miss` или crash;
- learning mode будет всё время находить новые «misses» в этих регионах.

**Возможные эвристики:**
1. Маркировать функции как «inline-data producer», если они начинаются
  с `PLA / PLA / STA $zp` (или похожего pattern-а). Для их
  call-site не добавлять `pc+3` в seeds.
2. Принимать список таких функций через `.cfg` (расширить формат:
  `inline_data_func=$XXXX`).
3. Для подозрительных regions (короткие «функции», заканчивающиеся
  невалидными опкодами или wraparound через `0xFFFF`) — проверять,
  не попадают ли они на JSR-target inline-pattern.

### 2.3. Jump tables возле `JMP (ind)` не парсятся

**Где:** ветка `mn == "JMP" and op.mode == "ind"` в `emit_instruction`
и отсутствие специальной логики в BFS.

**Что не так:** распространённый паттерн dispatcher-а:

```
LDA action_index
ASL A
TAX
LDA jump_table+1,X
PHA
LDA jump_table,X
PHA
RTS                 ; "return" в handler через стек
jump_table:
    .word handler_walk, handler_jump, handler_attack, ...
```

или более прямой:

```
JMP (handler_table)
handler_table:
    .word handler_a, handler_b, ...
```

Текущая реализация для `JMP (ind)` генерирует runtime-диспатч
через `cpu.PC = ind_addr(...)` — обработчики остаются невидимыми
для BFS. Они находятся только через learning mode, и каждый
вызов будет dispatch miss → интерпретатор → cfg-collection →
пересборка.

**Эвристика:** при встрече `JMP (abs_addr)` или цепочки
`LDA tbl,X / STA $zp / LDA tbl+1,X / STA $zp+1 / JMP ($zp)`
проверить, лежит ли `abs_addr` (или `tbl`) в PRG-ROM. Если да —
прочитать оттуда пары байт пока они выглядят как валидные
PRG-адреса (`>= $8000`, попадают в len(prg)), и добавить
каждую такую пару как seed.

Это сильно поднимает статическое покрытие для action/RPG-игр,
где основная логика — большие switch-таблицы action-handler-ов.

---

### 2.4. Self-modifying code и код в RAM не обрабатываются вообще

**Где:** BFS работает только над PRG (`addr >= 0x8000`).

**Что не так:** некоторые игры (Battletoads — известный пример,
ряд демосцен-роликов, некоторые звуковые движки типа FamiTracker)
копируют подпрограммы в RAM (`$0000–$07FF` или WRAM `$6000–$7FFF`)
и исполняют их оттуда. Это критично для:
- DMC IRQ-handler-ов, которые должны выполняться без задержек кэша;
- кода, который сам себя модифицирует (типа JIT-decompressor-а).

**Симптом:** при первом же `JSR $0500` или `JMP $0600` будет
dispatch miss → интерпретатор. Learning mode тут не поможет:
seed `$0500` бесполезен, потому что в момент рекомпиляции
PRG там ничего нет.

**Что можно сделать:** ничего особенного. Эти случаи в принципе
не статически-рекомпилируемые, интерпретатор-fallback — правильное
решение. Стоит явно задокументировать в README, что для таких
игр производительность не отличается от чистого интерпретатора.

---

### 2.5. Orphan-фаза: магическое число 3

**Где:** `Disassembler.discover`:

```python
for off in range(3):
    addr = next_pc + off
    ...
```

**Что не так:** константа 3 необоснована. Между подпрограммами
часто:
- 0 байт (RTS сразу после `RTS` начинается новая функция);
- 1–3 байта alignment-pad-а (компиляторы ca65 могут такое
  вставлять);
- 16+ байт inline-таблицы;
- сотни байт уровневых данных.

Если между функциями ровно 4 байта данных, orphan-фаза их
пропустит. Если 2 байта — найдёт. Случайно.

**Возможные улучшения:**
- увеличить окно до, скажем, 16 байт;
- параллельно валидировать «выглядит ли разобранное как
  правдоподобный prologue» (часто функции начинаются с
  `LDA/LDX/LDY/JSR/SEI/CLD/PHA`);
- принимать список orphan-seeds через `.cfg`.

---

### 2.6. Function-versus-block: yield после каждой инструкции

**Где:** базовая архитектура `_emit_full_c` — на функцию из BFS
эмитируется одна C-функция, и каждая инструкция внутри неё
заканчивается `g_cpu_cycles += N` + (на control flow) `return`.

**Что не так:** это не сам баг, а упущенная оптимизация.
Между двумя control-flow инструкциями (внутри basic block)
сейчас идёт N пар «прибавь циклы + продолжай». Можно было бы:

- группировать прямолинейный код в блоки;
- прибавлять циклы пачкой в конце блока;
- yield-ить PPU/APU только на границах блоков.

Это сильно ускорило бы выполнение. Возражение: некоторые
чтения внутри блока (например, `BIT $2002`) имеют тайминг-зависимый
эффект, и PPU должен быть «правильно тикнут» до этого момента.

**Компромисс:** базовые блоки с явным yield перед чтениями
`$2000–$2007`, `$4000–$401F`. Все остальные инструкции внутри блока
выполняются без промежуточных тиков, а циклы аккумулируются.

Текущий подход гарантирует точность за счёт производительности —
это валидный выбор, но стоит хотя бы измерить, насколько проигрывает
интерпретатору. Если проигрыш есть или выигрыш меньше 1.5× — смысл
всей затеи под вопросом.

## 3. Мелкие проблемы и стилистика

### 3.1. Лишнее присваивание `cpu.PC` перед каждой инструкцией

**Где:** `_emit_full_c`:

```python
for pc, op, operand in insns:
    lines.append(f' cpu.PC = 0x{pc:04X};')
    ...
```

**Проблема:** `cpu` — глобальная структура, компилятор не всегда
сможет оптимизировать мёртвые записи (особенно при `-O0/-O1` или
если cpu — `volatile`). Достаточно ставить PC только на границах
basic block-а (чтобы интерпретатор знал, откуда продолжить
при miss) и на control flow.

**Не блокер**, но измеримый overhead на горячих циклах.

---

### 3.2. Парсер ASM: edge cases в `.byte` / `.word`

**Где:** `parse_asm_labels`, ветки обработки `.byte`, `.db`,
`.word`, `.addr`, `.dw`, `.res`, `.ds`.

**Что не так:**
- строковые литералы считаются как `end = rest.find(quote, 1)` —
  не учитываются escape-последовательности (`\"`, `\\`);
- комбинированные `.byte "ABC", 0, $FF` посчитают только длину строки
  до первой запятой;
- макросы ca65 (`.macro`/`.endmacro`) не обрабатываются — содержимое
  макроса при инстанциировании увеличит PC, а парсер этого не учтёт;
- expression-ы типа `.byte (label_end - label_start)` дадут размер 1,
  но реально 1 байт — здесь как раз корректно, повезло.

**Симптом:** для сложных ca65-исходников некоторые labels могут
оказаться на неправильных адресах. Для простых .asm — работает.

**Решение:** либо принять, что для сложных asm-файлов нужны
ручные cfg, либо подключить полноценный ca65-parser (трудозатратно).

---

### 3.3. `.cfg` формат: нет `inline_data_func`, нет per-bank-seed

**Где:** `parse_cfg`.

**Что не так:** текущий формат поддерживает только `extra_func=$XXXX`
и `data_region=$XXXX,$YYYY`. Нет способа сказать:
- «эта функция читает inline-данные через PLA/PLA, не добавляй
  её call-site как seed» (см. 2.2);
- «эта функция в банке 5» (для будущего bank-aware рекомпиляции);
- «эта область — jump-table из N пар адресов» (для эвристики 2.3
  в ручном виде).

---

### 3.4. Учёт data_region из cfg на стороне discovery

**Где:** `parse_cfg` парсит `data_region`, но `Disassembler.discover`
эти регионы **не использует**. BFS-обход не проверяет,
не лежит ли текущий PC в data_region.

**Симптом:** ручная разметка data-регионов в cfg ничего не делает.
Пользователь может подумать «я указал data_region, должно сработать»,
но эффекта нет.

**Решение:** добавить проверку `addr in any data_region` в начале
обработки seed-а и при шаге BFS.

---

### 3.5. Обработка wrap-around на границе 0xFFFF

**Где:** `_bfs`:

```python
pc += op.size
if pc > 0xFFFF:
    break
```

**Что не так:** обрыв BFS при выходе за 0xFFFF — корректно.
Но на 6502 PC — 16-битный регистр, и `JMP $0000` (с переходом в RAM)
формально возможен. Текущий BFS не зайдёт в `$0000–$7FFF` (фильтр
`addr < 0x8000`), но `wrap-around` при последовательном
обходе (`JMP $FFFE` → дальше = `$0000`) — экзотический случай,
никто реально так не пишет, но стоит явно отметить, что **wrap
не поддерживается**.

---

### 3.6. Незначительная стилистика

- В `OPTABLE` некоторые `LAX zpy` имеют `size=2` — корректно,
  но проверьте, что недокументированный `STX zpy` (0x96)
  и `LDX zpy` (0xB6) различены от `STX zpx` правильно. (Бегло
  выглядит OK.)
- `_MORE_UNDOC` определяет `0xEB SBC #imm` как `SBC` — это
  правильно (это альтернативный legal-style opcode для SBC),
  но он попадёт в обычную ветку `mn == "SBC"` в эмиттере — что и нужно.
  То есть это **не** баг, в отличие от других undoc — но стоит
  убедиться, что для всех `_MORE_UNDOC` существуют либо ветки
  в эмиттере, либо явный отказ.

## Приоритеты

Если делать **в одну сессию**, в порядке окупаемости:

1. **1.1** (ADC/SBC двойной read) — 10 минут, реальный баг с
  потенциально серьёзными симптомами.
2. **1.3** (BRK PC+1 vs PC+2) — 5 минут.
3. **1.2** (undoc opcodes) — час, либо реализовать, либо удалить
  из OPTABLE.
4. **1.4 + 1.5** (page-cross и branch циклы) — пара часов, важно
  для всего тайминг-зависимого кода.
5. **2.3** (jump-table эвристика) — пара часов, сразу даст
  прирост покрытия для большинства игр.
6. Тестирование на NESTest — пара часов настройки
  (см. `nesrecomp-verification.md`).

**Перед инвестицией в bank-aware рекомпиляцию (2.1)** — сначала
замерить, насколько вообще проигрывают игры с большим объёмом
switchable-кода. Если разница в FPS между NROM и MMC3 невелика,
эта большая переделка не окупается, и текущая модель «fallback
на интерпретатор для switchable банков» — корректный и достаточный
ответ. Если разница значительна — тогда уже браться.

Также **проверить UNROM/MMC1-игры** — там потенциально проблема
не такого же сорта, как в MMC3 (fallback не сработает, потому что
`is_switchable` для них возвращает False). Если на практике
работает — модель валидна; если нет — нужно либо расширить
`is_switchable` на mapper 1/2 (и потерять статическое покрытие
в их switchable слоте), либо делать bank-aware.
