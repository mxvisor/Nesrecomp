# NESRecomp — верификация и полное покрытие

Методологический документ для проекта `mxvisor/Nesrecomp`. Описывает,
**как проверять корректность рекомпиляции** и **как добиваться полного
покрытия кода**. В отличие от документа `nesrecomp-bugs.md` (который
перечисляет конкретные баги текущего кода и их исправления), здесь —
критерии успеха, автоматизация сравнения с эмулятором, стратегии
нахождения всех адресов-целей и тестовый план.

Связанный документ: `nesrecomp-bugs.md` — баги и их исправление.

---

## Критерии успеха и автоматизация проверки

Цель проекта — **полная статическая рекомпиляция без багов с
геймплеем, идентичным оригиналу**. Это распадается на два
**независимых** измерения, которые нельзя смешивать:

1. **Корректность** — рекомпилированный код исполняет программу
  точно так же, как реальный 6502/2A03. Измеряется сравнением
  состояния с референсным эмулятором.
2. **Полнота рекомпиляции** — какая доля реально исполняемого кода
  проходит через сгенерированные C-функции, а не через
  интерпретатор-fallback. Измеряется счётчиком dispatch-miss.

Игра может быть полностью корректной, но иметь 70% полноты (30%
времени в интерпретаторе — это работает и выглядит идентично, но
это не «полная рекомпиляция»). И наоборот — 100% полноты с багом
в одном опкоде даст неверный геймплей. Поэтому нужны оба критерия
одновременно.

### Фундаментальное ограничение полноты

**Абсолютно полное статическое покрытие в общем случае
неразрешимо.** Косвенные переходы (`JMP (ind)`, RTS-trick,
computed jump tables) могут вести на адреса, вычисляемые в рантайме
из данных, которые статически неизвестны. Это вариант проблемы
остановки — нельзя статически доказать, что найдены все адреса,
куда управление может перейти при *любом* возможном вводе.

Поэтому практически достижимый критерий «полноты» переформулируется:

> **0 dispatch-miss на корпусе демок, покрывающем все достижимые
> игровые состояния.**

То есть полнота определяется относительно **корпуса FM2-демок**,
а не абсолютно. Чем полнее корпус (все уровни, все враги, все
ветки меню, экраны Game Over, концовка), тем ближе это к настоящей
полноте. Для игр с ограниченным числом состояний (Battle City,
SMB1) достижимо реальное 100%. Для больших RPG — практическое
покрытие «всего, что встречается в длинном прохождении».

Это нужно явно зафиксировать в README, чтобы «полная рекомпиляция»
не вводила в заблуждение: она полная **относительно
протестированного корпуса**.

### Шкала уровней успеха

| Уровень | Название | Критерий | Автоматизируется |
|---------|----------|----------|------------------|
| L0 | Boots | Запускается, выходит на главный экран, не падает за 600 кадров | Да |
| L1 | Playable | Прогон FM2-демки до конца без зависаний и crash | Да |
| L2 | State-equivalent | CPU+PPU-состояние идентично референсу на каждом семпле | Да (главное) |
| L3 | Static-complete | 0 dispatch-miss на корпусе демок | Да |
| L4 | Cycle-accurate | Проходят blargg cpu_timing / instr_timing | Да |

Целевой уровень для «без багов + идентичный геймплей» — **L2 + L3
одновременно**. L4 — желательный максимум (нужен после фиксов 1.4
и 1.5), но для многих игр геймплейная идентичность достигается и
без полной cycle-accuracy.

### Референсный эмулятор: Mesen

Mesen подходит лучше всего, потому что у него есть headless-режим
и Lua API для дампа состояния. Ключевые факты (по актуальной
документации):

- **Headless-запуск:** `Mesen --testrunner <game> <script.lua>` —
  грузит игру и скрипт, исполняет на максимальной скорости до
  вызова `emu.stop(exitCode)`. Exit code возвращается процессом —
  это и есть pass/fail для CI.
- **Состояние:** `emu.getState()` возвращает таблицу. Для NES в
  ней (как минимум) `cpu = { pc, a, x, y, sp, status, cycleCount,
  nmiFlag, irqFlag }` и `ppu = { cycle, scanline, frameCount,
  control{...}, status{...}, state{...} }`, плюс `apu`.
- **Память:** `emu.read(addr, emu.memType.<...>)` читает CPU-RAM,
  PPU-RAM, OAM, PRG/CHR ROM/RAM и т.д.
- **Хуки:** `emu.addMemoryCallback(fn, type, start, end)` для
  exec/read/write, `emu.addEventCallback(fn, eventType)` для
  startFrame/endFrame/nmi/irq.

**Важная оговорка по версиям:** документация `emu.getState()` на
mesen.ca написана для Mesen 0.9.9 и **не полностью совместима с
Mesen 2**. В Mesen 2 `getState()` возвращает суперсет старых полей,
но точный состав не документирован — нужно один раз дампнуть
таблицу (рекурсивным обходом с `emu.log`) и зафиксировать реальные
имена полей для используемой версии. Делать это до написания
скриптов сравнения.

### Автоматизация L0 (Boots)

Lua-скрипт для Mesen, который считает кадры и проверяет, что игра
дошла до известного состояния (например, ненулевой scanline-вывод
или конкретный байт в RAM, выставляемый после инициализации):

```lua
-- boot_check.lua (reference behaviour для Mesen)
local frames = 0
local function onEndFrame()
  frames = frames + 1
  if frames >= 600 then  -- ~10 секунд NTSC
    -- проверка: PPU rendering включён (игра что-то рисует)
    local st = emu.getState()
    if st.ppu.control.backgroundEnabled or st.ppu.control.spritesEnabled then
      emu.stop(0)  -- PASS
    else
      emu.stop(1)  -- FAIL: чёрный экран
    end
  end
end
emu.addEventCallback(onEndFrame, emu.eventType.endFrame)
```

Для рекомпилированной версии — аналогичная проверка средствами
самого runner-а (счётчик кадров + флаги PPUMASK), возврат exit code.
В CI: запустить оба, сверить exit codes.

### Автоматизация L2 (State-equivalent) — основной механизм

Это сердце проверки «идентичности оригиналу». Два под-метода,
от грубого к точному.

**Метод A — пофреймовый хэш (быстрый, для длинных демок).**

На каждом N-м кадре (например каждом) берётся хэш от среза
состояния: CPU-RAM (`$0000–$07FF`), PPU nametable/OAM/палитра,
ключевые CPU-регистры. Сравнивается с эталоном. Первое расхождение
по кадру = точка дивергенции.

```lua
-- frame_hash_ref.lua (Mesen reference)
local out = io.open("ref_hashes.txt", "w")
local function fnv1a(bytes)         -- простой 32-битный хэш
  local h = 2166136261
  for _, b in ipairs(bytes) do
    h = (h ~ b) & 0xFFFFFFFF
    h = (h * 16777619) & 0xFFFFFFFF
  end
  return h
end
local function onEndFrame()
  local st = emu.getState()
  local bytes = {}
  for a = 0, 0x7FF do bytes[#bytes+1] = emu.read(a, emu.memType.nesMemory) end
  -- добавить OAM/палитру/нужные PPU-поля по необходимости
  out:write(string.format("%d %08X %04X\n", st.ppu.frameCount, fnv1a(bytes), st.cpu.pc))
  out:flush()
end
emu.addEventCallback(onEndFrame, emu.eventType.endFrame)
```

Runner рекомпилированной игры пишет файл того же формата
(`recomp_hashes.txt`). Сравнение:

```bash
diff <(cut -d' ' -f2 ref_hashes.txt) <(cut -d' ' -f2 recomp_hashes.txt) \
  | head -1   # первый разошедшийся кадр
```

Хэш CPU-RAM + OAM ловит почти любую геймплейную дивергенцию
(позиции, счёт, состояние врагов), но не ловит расхождения,
которые «самоисправляются» внутри кадра. Для геймплейной
идентичности этого почти всегда достаточно.

**Метод B — пошаговый trace-diff (строгий, для отладки CPU).**

Самый строгий критерий корректности: сравнение CPU-состояния на
**каждой инструкции** в формате NESTest-лога:

```
C5F5  A9 00     LDA #$00    A:00 X:00 Y:00 P:24 SP:FD CYC:12345
```

Mesen умеет это через Trace Logger (или через
`emu.addMemoryCallback(..., emu.memCallbackType.cpuExec)` с дампом
`emu.getState().cpu`). Runner должен уметь печатать тот же формат
(текущий эмит ставит `cpu.PC` перед каждой инструкцией — баг 3.1,
но для trace-режима это как раз удобно; можно сделать
`#ifdef TRACE`).

```bash
# обрезать оба лога до релевантных полей и найти первое расхождение
diff ref.trace recomp.trace | head -20
```

Это инструмент отладки: когда метод A показал «дивергенция на
кадре 1234», запустить trace-diff и найти **точную инструкцию**,
где состояние впервые разошлось. Обычно это сразу указывает на
баг (например, неучтённый page-cross цикл сдвинул ветвление, или
двойной `mem_read` из бага 1.1).

**Нюанс синхронизации:** trace-сравнение требует, чтобы обе стороны
стартовали из идентичного состояния (power-on state) и получали
идентичный ввод покадрово. Ввод подаётся либо воспроизведением
одной и той же FM2-демки, либо программно из Lua по таблице
`кадр → байты кнопок` (та же таблица — в runner). Power-on RAM
должна инициализироваться одинаково (см. ниже).

### Автоматизация L3 (Static-complete)

В dispatch-таблице runner-а ветка `default:` (уход в
`cpu_interp_step`) инкрементирует счётчик с разбивкой по адресам:

```c
// в runner.c, в default-ветке dispatch
miss_count++;
miss_addr_histogram[cpu.PC]++;   // какие адреса чаще всего мимо
```

После прогона корпуса демок:

```
static_coverage  = (байты PRG, попавшие в BFS) / (всего исполнимых байт PRG)
executed_coverage = инструкций через func_* / всего исполнено инструкций
miss_rate         = miss_count / total_dispatch_calls
```

**Критерий L3:** `miss_rate == 0` на всём корпусе. Если не ноль —
гистограмма `miss_addr_histogram` показывает топ-адреса, куда
уходит интерпретатор. Эти адреса скармливаются обратно в `.cfg`
как seed-ы (это и есть learning mode), и BFS пересобирается.
Цикл повторяется, пока miss_rate не станет нулём.

Для NROM/CNROM (Battle City, SMB1, Adventure Island) ожидается
честный 0. Для MMC1/UNROM/MMC3 — 0 только после решения вопроса
2.1 (bank-aware рекомпиляция); до этого miss_rate отражает долю
кода в switchable банках.

### Автоматизация L4 (Cycle-accurate)

blargg-овские test ROM (`cpu_timing_test6`, `instr_timing`)
сами выводят результат на экран и в `$6000` (статус-байт). Lua
читает `$6000`: `0x00` = passed, остальное = код ошибки. Полностью
автоматизируется тем же `--testrunner` + `emu.stop(status)`.

### Детерминизм: обязательное условие любого сравнения

Чтобы сравнение вообще было осмысленным, обе стороны должны быть
**побитово детерминированы** от одинаковых начальных условий:

- [ ] **Power-on RAM.** Реальная NES имеет неинициализированную RAM.
  Mesen по умолчанию заполняет её паттерном; runner должен
  использовать **тот же** паттерн (настроить Mesen на all-zeros
  или повторить его дефолтный паттерн в runner). Несовпадение
  здесь даст ложные дивергенции в играх, читающих
  неинициализированную память.
- [ ] **Идентичный ввод покадрово.** Одна FM2-демка для обеих
  сторон, либо общая таблица `кадр → кнопки`.
- [ ] **Открытая шина (open bus).** Чтение неподключённых адресов
  должно вести себя одинаково (последнее значение на шине).
- [ ] **APU/DMC тайминги** при включённом сравнении APU-состояния.

### Сводный CI-пайплайн

```
для каждой игры из тестовой таблицы:
  1. recomp: nesrecomp.py game.nes  → game_full.c
  2. компиляция runner + game_full.c → game_recomp
  3. L0: mesen --testrunner game.nes boot_check.lua;  game_recomp --boot-check
         сверить exit codes
  4. L2-A: mesen --testrunner game.nes frame_hash_ref.lua + demo.fm2 → ref_hashes
           game_recomp --play demo.fm2 --hash-out → recomp_hashes
           diff → первый расходящийся кадр (ожидается: нет)
  5. L3: game_recomp --play corpus/*.fm2 --miss-stats → miss_rate
         (ожидается: 0)
  6. при расхождении на шаге 4 → автоматически запустить L2-B
     (trace-diff) вокруг расходящегося кадра для локализации
```

Каждый шаг возвращает exit code; весь пайплайн — обычный
`make test`, годный для GitHub Actions.

## Стратегии достижения полного покрытия

Главная трудность статической рекомпиляции 6502 — **разделение
кода и данных** и нахождение целей косвенных переходов
(`JMP (ind)`, RTS-trick, computed jump tables). Ниже — систематизация
способов нахождения адресов-целей, их надёжность и защита от
ложных seed-ов.

### Фундаментальный предел

**Абсолютно полное покрытие чисто статическими методами в общем
случае неразрешимо** — это не инженерное, а математическое
ограничение (сводится к проблеме остановки / теореме Райса). Если
адрес перехода вычисляется в рантайме из данных, зависящих от
ввода, никакой статический анализ не докажет, что найдены все
возможные значения.

На практике это подтверждается: статический анализатор Ghidra
находит в Super Mario Bros. лишь несколько десятков функций,
потому что не распознаёт RTS-trick jump engine без
runtime-информации. Из этого следует главный вывод:

> **Настоящие 100% достижимы только гибридом: статическая разметка
> (в идеале — byte-perfect дизассембли) + динамическая верификация
> на корпусе демок.**

### Источники адресов-целей и уровень доверия

| Источник | Тип | Доверие | Риск ложного seed |
|----------|-----|---------|-------------------|
| BFS от reset/NMI/IRQ-векторов | статический | высокое | низкий |
| Byte-perfect дизассембли (asm) | статический (ручная разметка) | **наивысшее** | минимальный |
| Динамический промах (live / FM2) | динамический | **наивысшее** | почти нулевой |
| CDL из Mesen/FCEUX | динамический | высокое | почти нулевой |
| Jump-table эвристика (скан `.word`) | статический (эвристика) | **низкое** | высокий |
| Orphan-скан между функциями | статический (эвристика) | низкое | средний |

Два источника дают практически безошибочные seed-ы — **byte-perfect
asm** и **динамический trace**. Эвристики (jump-table scan,
orphan-скан) требуют верификации.

### Byte-perfect дизассембли как ground truth

Если доступен полный дизассембли, который реассемблируется в
**идентичный ROM** (совпадение по SHA-256/CRC), — это наилучший
из возможных источников. В таком проекте каждый байт уже
классифицирован человеком как код или данные, и все jump-таблицы
размечены явно:

```asm
JumpEngine:
    asl                      ; index * 2
    tay
    lda JumpTable+1,y
    pha
    lda JumpTable,y
    pha
    rts                      ; "переход" через стек
JumpTable:
    .word Handler_Walk       ; ← эти метки — code-targets,
    .word Handler_Jump       ;    которые иначе были бы промахами
    .word Handler_Attack
```

Все цели computed jumps здесь уже извлечены человеком. Это
снимает главную нерешаемую статически проблему.

**Критический нюанс парсинга.** Чтобы реально получить эти targets,
`parse_asm_labels` должен **интерпретировать `.word`/`.addr`/`.dw`
таблицы как списки code-target-ов**, а не просто как данные.
Сейчас парсер считает их размер для вычисления адресов меток, но
не добавляет содержимое таблицы как BFS-seed-ы. Нужно:

- [ ] Распознавать `.word`/`.addr`/`.dw` рядом с меткой-диспетчером
  (или помеченные в asm-комментарии как jump table) и добавлять
  каждый адрес из таблицы как seed.
- [ ] Опционально — расширить cfg-формат: `jump_table=$XXXX,N`,
  чтобы вручную указывать таблицу из N адресов там, где asm
  неоднозначен.

Информация о целях в byte-perfect asm присутствует — задача
рекомпилятора в том, чтобы её не потерять при парсинге.

#### Доступные дизассембли для тестовых игр

Для тестового корпуса byte-perfect дизассембли в основном
существуют (имена проектов на момент написания, проверять
актуальность перед использованием):

- **SMB1** — дизассембли doppelganger'а (часто форкается как
  `smb1`), требует CHR-данные и iNES-заголовок для сборки.
- **SMB2** — `smb2`-проекты со сборкой в byte-for-byte копию,
  верификация по SHA-256.
- **SMB3** — `smb3`-проект, реассемблируется в идентичный
  SMB3 US (PRG1) через NESASM.
- **Mario Bros.** — byte-accurate дизассембли (NTSC/PAL/GC),
  собирается под cc65/asm6.

Для Battle City, Zelda 1, Mega Man, Adventure Island наличие
полного byte-perfect дизассембли надо проверять отдельно; при
отсутствии — полагаться на динамический trace + эвристики
с верификацией.

### Поведение при ложном seed

Ключевой принцип: **ложный seed сам по себе не нарушает
корректность, пока dispatch остаётся точным.** Если адрес на
самом деле данные, runtime никогда не прыгнет на него как на код,
сгенерированная «функция» никогда не вызовется — это мёртвый код.
Издержки только в раздутом `_full.c` и времени компиляции.

Опасность возникает в двух случаях:

1. **Bank-коллизия.** Ложный адрес в switchable слоте совпал с
  реальным кодом другого банка → dispatch может вызвать не ту
  функцию. Это проявление проблемы 2.1.
2. **Каскад.** Мусорная «функция» из данных содержит `JMP`/`JSR`
  в случайные адреса → BFS тянет ещё больше ложных seed-ов,
  замусоривая граф (родственно inline-data проблеме 2.2).

### Защита от ложных seed-ов

- [ ] **Разделять доверие по источнику.** Помечать каждый seed
  его происхождением (`vector` / `asm` / `dynamic` / `cdl` /
  `heuristic`). Seed-ы из эвристик обрабатывать отдельно.
- [ ] **Cross-validation эвристических seed-ов.** Seed из
  jump-table/orphan-скана подтверждать динамическим прогоном:
  если за весь корпус демок runtime ни разу туда не прыгнул —
  пометить подозрительным, не включать в финальную сборку.
- [ ] **Byte-perfect asm имеет приоритет.** При конфликте «asm
  говорит данные, эвристика говорит код» — верить asm.
- [ ] **Sanity-фильтр на разобранную функцию.** Отбрасывать
  кандидатов, которые содержат невалидные опкоды, не имеют
  терминатора (RTS/RTI/JMP), уходят в wraparound через `0xFFFF`
  или начинаются с явно нелогичного опкода.
- [ ] **L2 state-equivalence — финальный арбитр.** Ложный seed
  не нарушит идентичность эмулятору, если функция не вызывается.
  Если игра побайтово совпадает с Mesen на всём корпусе — cfg
  может быть «грязным», но корректность сохранена.

### Рекомендуемая стратегия для настоящих 100%

Порядок применения источников, от самого надёжного:

1. **Если есть byte-perfect дизассембли** — использовать его как
  основу разметки код/данные и источник всех jump-таблиц. Это
  снимает нерешаемую статически часть задачи.
2. **BFS от векторов** — базовое статическое покрытие достижимого
  напрямую кода.
3. **Динамический прогон корпуса демок** (FM2 в headless) —
  верификация полноты: всё достижимое реально исполняется,
  0 dispatch-miss (критерий L3).
4. **Эвристики (jump-table, orphan)** — только для игр без
  готового дизассембли, и обязательно с cross-validation
  против динамики.
5. **L2 state-equivalence** — финальная проверка корректности
  всего, что собралось.

Комбинация (1) + (3) с верификацией (5) даёт практические 100%
для конкретной игры. Чисто автоматический путь без дизассембли
((2) + (4) + (3)) приближается к 100%, но гарантию даёт только
полнота корпуса демок плюс отсутствие непокрытых computed jumps.
